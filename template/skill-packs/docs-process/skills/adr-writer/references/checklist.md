# ADR Quality Checklist

Use this checklist when writing or reviewing an Architecture Decision Record.

## Structure

- [ ] Sequential ADR number assigned and consistent with existing records.
- [ ] Date and status (`proposed` / `accepted` / `deprecated` / `superseded`) present.
- [ ] Deciders or accountable roles listed.

## Problem Statement

- [ ] States the question or tension clearly without embedding the answer.
- [ ] Lists driving forces: requirements, constraints, risks, and quality attributes.
- [ ] Readable by someone who does not already know the outcome.

## Options

- [ ] At least two realistic options presented (no straw-man entries).
- [ ] Each option has concrete pros, cons, and relevant evidence.
- [ ] Evidence is verifiable or explicitly marked as an estimate.

## Decision & Consequences

- [ ] Chosen option is stated explicitly with primary rationale.
- [ ] Positive consequences linked back to driving forces.
- [ ] Negative consequences and accepted risks documented.
- [ ] Follow-up work or open questions captured where applicable.

## Consistency

- [ ] No contradictions with currently accepted ADRs; superseded records updated.
- [ ] Cross-references to related ADRs, RFCs, or design docs included where relevant.
- [ ] Follows project directory convention and filename pattern.
